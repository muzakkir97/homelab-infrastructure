### 2026-07-23 - Kinmoon Hard Drive 1 failure: do not attempt RAID rebuild until physical replacement
**Decision:** Do not click "Repair" on Kinmoon's Storage Pool 1 (RAID 1) until Hard Drive 1 is physically replaced with a healthy drive. Root cause of Hard Drive 1 failure: genuine progressive media failure, evidenced by Reallocated Sector Count trending March 573 → April 609 → May 1,963, with current SMART Index 5 at 133 (below failure threshold 140). UGOS RAID 1 pool status shows Degraded; Hard Drive 1 SMART status Critical. Current operational state: RAID 1 running on single healthy drive (Hard Drive 2) with zero redundancy. Risk: if Hard Drive 2 fails before replacement, entire proxmox-backups offsite backup archive (2.4TB, only downstream copy) would be lost. Correct action sequence: (1) source and physically replace Hard Drive 1, (2) then trigger UGOS Storage app Repair function to rebuild the mirror. Doing Repair now would either be a no-op against a failing drive or add further read/write stress to Hard Drive 1, risking acceleration of the failure.
**Why:** RAID rebuild (triggered by Repair) requires a healthy second member to rebuild onto. Attempting it against Hard Drive 1 in its current degraded state violates the principle of "diagnose before acting" — SMART data already gives a clear hardware diagnosis without requiring a destructive test. This matches the successful pattern used on Kuromoon's hdd-backup-1 SATA issue (2026-07-14), where investigation led to a specific root cause that could be acted on precisely rather than blindly.
**Alternatives considered:** Click Repair immediately to see if the pool self-heals (rejected — SMART data already shows progressive media failure, not a transient fault that Repair can resolve), replace the drive before investigating (rejected — understanding the failure mode prevents repeat hardware selection errors), rebuild onto Hard Drive 2 by converting to RAID 0 (rejected — would delete all backup history; defeats the purpose).

### 2026-07-23 - Kinmoon-smb NAS recycle bin cleanup deferred pending Hard Drive 1 replacement
**Decision:** Defer the kinmoon-smb NAS-level recycle bin toggle/cleanup task (`#recycle` folder, 1.3TB) in favor of prioritizing the Hard Drive 1 replacement first. Current impact of recycle bin: 1.3TB of files already logically freed by Proxmox's `prune-backups` retention policy (keep-daily=7, keep-weekly=4) are double-counted in NAS storage metrics, making the proxmox-backups share report 94% full (2.5TB used / 2.7TB capacity) when actual useful backup data is only ~1.3TB. This is a space-pressure issue, not a data-safety issue. Hard Drive 1's failure is a live risk to backup redundancy. Proxmox retention policy is confirmed working correctly; the recycle bin is cosmetic overhead.
**Why:** Addressing backup redundancy (Hard Drive 1 replacement) prevents data loss; clearing the recycle bin only improves headroom. Both are now tracked as separate action items; Hard Drive 1 takes priority sequentially. The recycle bin cleanup is straightforward once prioritized (likely via UGOS Storage app's per-share settings, path not yet fully mapped), but is not blocking any critical operation.
**Alternatives considered:** Perform both cleanup and hardware investigation in the same session (rejected — user elected to close out session after Hard Drive 1 discovery and summarize findings), ignore the recycle bin entirely (rejected — 1.3TB unused space is worth recovering once Hard Drive 1 is resolved and NAS capacity confidence is restored).

### 2026-07-23 - Prometheus "CPU usage" alert query: iowait confusion flagged for future refinement
**Decision:** Flagged Prometheus "CPU usage" alert rule query for future refinement, but no alert rule changes made this session. Root finding: the alert query sums all non-idle CPU time, which includes iowait (`%wa` in `top`) as well as genuine compute usage (`us` and `sy`). CT 205 (Alertmanager) "CPU usage 100%" alert on 2026-07-20 03:46 was confirmed via `top` to show 100.0% `%wa` with 0.0% `us`/`sy` — pure I/O wait, zero actual compute load. Correlated to nightly `backup-daily` vzdump job (runs 02:00-~03:35, writing to kinmoon-smb CIFS share at 94% capacity) causing lingering I/O pressure past nominal job completion. Recommendation: update the alert rule to either exclude iowait from the calculation or split into separate `CPUUsage` and `IOWaitUsage` alerts to distinguish compute-bound from I-O-bound congestion.
**Why:** The current rule conflates two different operational states that have different remediation strategies. A "CPU usage 100%" alert triggers investigation of container compute allocation and process load; an "I/O wait 100%" alert triggers investigation of storage/network latency. Alerting on iowait as "CPU usage" is technically correct but operationally confusing and leads to misdirected troubleshooting (e.g., considering CT 205 container resource increases when the actual issue was a network share at 94% capacity).
**Alternatives considered:** Silence the alert rule on future iowait-related triggers (rejected — the alert is legitimate, just mislabeled; silencing masks signal), immediately update the alert rule this session (rejected — this was a diagnosis session, not an alert-rule refactor; the rule change deserves its own focused work session), treat "CPU usage" and iowait as the same (rejected — operationally distinct and require different responses).

### 2026-07-17 - Pelican web file editor banned for PalWorldSettings.ini; pct exec standardized
**Decision:** Abandoned Pelican's web-based file editor for all future PalWorldSettings.ini edits. Standardized on `pct exec` + `sed`/direct file writes from the Proxmox host instead. Root cause: Pelican's web editor introduces hard line breaks into PalWorldSettings.ini's OptionSettings=(...) block when saving edits (corrupted file observed: 13 lines instead of required 2). Palworld silently ignores the entire OptionSettings block when the block is broken across multiple lines, falling back to defaults for all settings in that block (confirmed: PalCaptureRate change had no in-game effect despite correct file content). Safe editing method: `pct exec <CTID> -- sed -i 's/old/new/g' /path/to/PalWorldSettings.ini` or equivalent direct file writes from Proxmox host. This bypasses Wings and Pelican's file editor entirely. Related: Pelican's PalworldServerConfigParser (run at every container boot) auto-populates PublicIP from local allocation (192.168.30.219) when the egg's "Public IP" variable is empty, causing silent reverts of manual edits; fixed by enabling User Editable + User Viewable permissions on the egg variable and populating it at container creation time.
**Why:** Pelican's web editor is not safe for multi-line ini values. Direct pct exec access to the container filesystem is reliable, preserves file formatting, and bypasses the buggy editor entirely. Prior lesson (avoiding ini edits via Pelican) was now reconfirmed with root-cause detail. This is a documented constraint for all Pelican-managed servers with complex ini structures going forward.
**Alternatives considered:** Delete and regenerate PalWorldSettings.ini from a template (rejected — loses valid edits already made), use Pelican's web editor more carefully (rejected — the tool itself corrupts the format; careful use doesn't prevent the bug), edit via SSH directly (rejected — less reliable than pct exec for container access from Proxmox host).

### 2026-07-17 - Palworld backup method: pct exec + pct pull standardized; panel download bypassed

Workaround is proven reliable and doesn't require debugging the panel. Palworld SaveGames backup created this session: `/mnt/hdd-backup-1/palworld-backup-20260716/palworld-save-backup.tar.gz` (22M, verified contents).
**Why:** Panel download is broken; direct pct access works every time. For live data that needs immediate backup (player progress at stake), the pct method is faster than debugging signed URLs. This same method applies to any Pelican container on any node, not just Palworld.
**Alternatives considered:** Continue debugging the 404 error (rejected — time-sensitive backup need took priority, and pct method is proven), try different browsers/caching (rejected — issue reproduced even on internal IP, ruling out Cloudflare), request Wings/panel logs for signed URL generation (rejected — pct method works now, can investigate panel issue later if needed).
