### 2026-07-09 - Funnel Agent design principle established; MERLIN/Midas re-sourced from existing infra
**Decision:** Established "Funnel Agent" classification: an agent is a funnel if there is existing infrastructure it sits on top of and translates into a daily digest with a recommended action. Test: "If there's infra that could be deployed and the agent sits on top and translates it, that agent is a funnel." Classification as of July 9, 2026: MERLIN (Uptime Kuma), Midas (Langfuse), Cu Chulainn (Alertmanager, once built) are funnels; Jeanne Alter, Da Vinci, Scathach are not; EMIYA is hybrid ("funnel-plus" — reads infra like a funnel but also executes on approval). Specific re-sourcing decisions: (1) MERLIN's SSL check now sources from Uptime Kuma's native certificate monitoring instead of direct Cloudflare API (resolves July 14 urgency, removes broken token dependency). (2) Midas's Haiku/Claude costs now source from Langfuse Metrics API instead of duplicate `gilgamesh_costs` Data Table; Ollama tracking stays in Data Table since those calls are $0 and outside Langfuse's automatic cost-inference model. (3) Funnel agents must report a recommendation ("what should I do about it"), not just raw status — this is now a required output field for all funnel implementations going forward.
**Why:** Prevents agents from duplicating effort already solved by dedicated tools (Uptime Kuma, Langfuse, Alertmanager), eliminates drift risk between duplicate data sources (MERLIN's hardcoded SSL date vs. Uptime Kuma's real-time cert tracking; Midas's Data Table costs vs. Langfuse's automatic tracking), and keeps agent build time focused on the actual differentiator — aggregation and actionability — rather than re-implementing plumbing that's already deployed. "What happened + what to do" pattern is the value; the underlying check is commodity.
**Alternatives considered:** Treat all agents uniformly with no funnel/non-funnel distinction (rejected — led directly to MERLIN and Midas duplicating Uptime Kuma and Langfuse functionality instead of reading it), build MERLIN/Midas from scratch without leveraging existing infra (rejected — wastes build time and creates maintenance burden), report raw numbers without recommendations (rejected — incomplete implementation of what makes a funnel valuable).

### 2026-07-09 - Scathach build requires LangGraph evaluation pre-implementation
**Decision:** Scathach (Lancer-class servant, career growth/research agent, 1st build priority in agent roster as of May 16, 2026) must have LangGraph evaluation completed before implementation scope is defined. Career-research workflows expected to require autonomous multi-step reasoning loops that n8n handles poorly for this use case. LangGraph fit assessment is a pre-requisite for Scathach's build scope definition, not optional pre-work. Updated sequencing (July 9, 2026): LangGraph evaluation should happen *after* Phase 24.12 (Jeanne Alter's refactor to n8n's native AI Agent node), not standalone — 24.12 is already testing whether n8n's native AI Agent node can handle multi-step autonomous reasoning for Jeanne Alter, and if it can, that same node may cover Scathach's career-research needs too, avoiding a second orchestration framework. Evaluation still required before Scathach implementation, just sequenced behind 24.12 rather than done independently.
**Why:** Current agents (Jeanne Alter, Da Vinci, MERLIN, Midas, EMIYA) fit n8n's strengths (multi-step workflows with external tools, stateful branching). Scathach's intended function (career research with multi-step autonomous reasoning) may require state-graph structure that LangGraph provides better than n8n, or it may turn out n8n's AI Agent node is sufficient — 24.12 provides the test case. Evaluating LangGraph early (but sequenced behind 24.12) prevents mid-build discovery that the chosen framework is mismatched to the use case.
**Alternatives considered:** Build Scathach on n8n first, refactor to LangGraph if needed (rejected — wastes time if mismatch is found), skip LangGraph evaluation entirely (rejected — risks building on the wrong framework), use CrewAI instead (rejected — already designated CrewAI for multi-agent discussion, not single-agent reasoning).

### 2026-07-08 - Multi-agent discussion protocol uses CrewAI, not LangGraph
**Decision:** Selected CrewAI as the target framework for future multi-agent discussion capability, superseding an earlier tentative LangGraph approach. Scoped as research priority, deferred from active development. Discussion trigger: complexity-based (quick topics via Telegram, complex/research topics run in background). Disagreement resolution: Jeanne Alter presents both sides; user arbitrates (explicitly not consensus-forced, not auto-decided). Reporting: summary-only by default, full transcript on request.
**Why:** CrewAI's role-based mental model (agents as distinct personas with roles and tools) maps directly onto the existing FGO-persona structure of Chaldea agents, making adoption faster and more coherent than LangGraph's lower-level state-graph approach. CrewAI gets to a working prototype faster. Human-in-the-loop arbitration preserves user agency over agent decisions and avoids the risk of agents forming a consensus that contradicts user intent.
**Alternatives considered:** LangGraph (rejected — more flexible but requires more custom state machine code, steeper learning curve for marginal benefit), multi-turn Jeanne Alter conversation (rejected — single-agent reasoning not suitable for distinct perspectives), force consensus (rejected — removes user arbitration, bad precedent for agent autonomy).
