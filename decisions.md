### 2026-07-08 - Ecosystem renamed from Kuromoon to Chaldea
**Decision:** Renamed the agents ecosystem layer from "Kuromoon" to "Chaldea". Kuromoon now refers exclusively to the physical hardware/homelab infrastructure. Chaldea designates the coordinated agent system layer (Jeanne Alter, Da Vinci, MERLIN, Midas, EMIYA, Guardian, Scathach, Solomon).
**Why:** Kuromoon was overloaded — referring to both physical hardware and the agent ecosystem caused persistent confusion in documentation and sessions. Chaldea stays within FGO/Fate lore (Chaldea is the organization that houses and coordinates Servants in canon), matching the ecosystem's actual role as a coordinated multi-agent system. Clear namespace separation improves documentation clarity and conversation consistency.
**Alternatives considered:** Keep Kuromoon for both (rejected — semantic overload), rename hardware instead (rejected — Kuromoon is well-established in physical infra docs), use generic term like "Agent Layer" (rejected — loses thematic consistency with existing agent naming).

### 2026-07-08 - Multi-agent discussion protocol uses CrewAI, not LangGraph
**Decision:** Selected CrewAI as the target framework for future multi-agent discussion capability, superseding an earlier tentative LangGraph approach. Scoped as research priority, deferred from active development. Discussion trigger: complexity-based (quick topics via Telegram, complex/research topics run in background). Disagreement resolution: Jeanne Alter presents both sides; user arbitrates (explicitly not consensus-forced, not auto-decided). Reporting: summary-only by default, full transcript on request.
**Why:** CrewAI's role-based mental model (agents as distinct personas with roles and tools) maps directly onto the existing FGO-persona structure of Chaldea agents, making adoption faster and more coherent than LangGraph's lower-level state-graph approach. CrewAI gets to a working prototype faster. Human-in-the-loop arbitration preserves user agency over agent decisions and avoids the risk of agents forming a consensus that contradicts user intent.
**Alternatives considered:** LangGraph (rejected — more flexible but requires more custom state machine code, steeper learning curve for marginal benefit), multi-turn Jeanne Alter conversation (rejected — single-agent reasoning not suitable for distinct perspectives), force consensus (rejected — removes user arbitration, bad precedent for agent autonomy).

### 2026-07-08 - Nextcloud Deck falls under Da Vinci's exclusive write scope
**Decision:** Nextcloud Deck (Homelab board, ID 4) is added to Da Vinci's documentation write scope, subject to the same "sole writer" discipline as Obsidian. No other agent (including Jeanne Alter, MERLIN, Midas) writes to Deck directly; all board updates flow through Da Vinci's pipeline.
**Why:** Preserves the single-writer-per-storage-system pattern that prevents concurrent write conflicts and maintains clear data provenance. Deck is a kanban view into the same ROADMAP/phase data that Da Vinci already manages, not a separate system. Centralizing writes eliminates merge conflicts and ensures board state stays synchronized with documentation.
**Alternatives considered:** Allow agents to write cards directly (rejected — introduces concurrent-write race conditions, breaks sync), use Deck as read-only reporting surface (rejected — defeats the purpose of a kanban board for work tracking), delegate Deck management to a separate agent (rejected — adds unnecessary coordination overhead).

### 2026-07-08 - Da Vinci documentation pipeline gains 9th step: Nextcloud Deck sync
**Decision:** Da Vinci's update pipeline gains a 9th step that syncs all documentation changes to Nextcloud Deck cards. Trigger: every pipeline run (broadest option). Matching mechanism: hidden `sync-id` tag in card description footer (format: `---\n🔖 sync-id: phase-24.10`). Update behavior: match-and-update existing cards silently for status changes; create new cards only for genuinely new items. Stack routing: status-keyword based (e.g., "Complete" → Done, "In Progress" → In Progress). Scope: Homelab board only (Career/Personal boards excluded for now). Precondition: all ~30 existing Homelab board cards must be manually backfilled with sync-id tags by the user before this automation goes live.
**Why:** Deck sync keeps the kanban board as a real-time view of work state without manual duplication. Broadest-trigger option chosen deliberately despite board-noise risk — better to over-sync than create blind spots where documentation updates silently miss the board. Hidden sync-id tags enable reliable parsing without cluttering human-readable card bodies. Manual backfill chosen over automated "adopt" pass because known board conflicts (duplicate Phase 7E cards, stale completed items still visible, Da Vinci Stage 2 status ambiguity) require human judgment; fuzzy automated matching could create worse conflicts.
**Alternatives considered:** Sync only certain phases (rejected — creates blind spots), always create new cards (rejected — duplicates on every status update), automated backfill (rejected — can't handle known conflicts correctly), store sync-id in card title (rejected — clutters readability).

### 2026-07-08 - Jeanne Alter architecture refactor targets MCP, Mem0, n8n AI Agent node, personality-file pattern
**Decision:** Scoped the planned Jeanne Alter architecture refactor around four proven external components: MCP (Model Context Protocol) for tool execution ("hands"), Mem0 (self-hosted, Qdrant-backed) for persistent memory, n8n's native AI Agent node for orchestration ("skeleton"), and a dedicated personality-file pattern for voice/persona. Explicitly rejected full re-platforming onto OpenClaw or Hermes Agent as alternative runtimes — maintains DIY approach on n8n for portfolio/resume value, but integrates proven external components rather than building every layer bespoke.
**Why:** Current architecture (hand-built If/branch logic, split between Data Table short-term + Qdrant RAG long-term, no standard tool protocol) is functional but brittle. MCP is becoming a protocol standard for agent tools (supported in n8n natively); Mem0 consolidates fragmented memory into one system. n8n's native AI Agent node reduces custom logic. Personality-file pattern (separate from system prompt) makes voice/persona updates separable from core logic. DIY on n8n preserves hands-on learning and portfolio value while reducing homegrown code per component.
**Alternatives considered:** Full migration to OpenClaw (rejected — loses n8n familiarity, harder to customize), migration to Hermes Agent (rejected — same issue), stick with current hand-built architecture (rejected — accumulating technical debt), migrate to LangChain (rejected — less suitable for n8n integration).

### 2026-07-08 - Universal time/date awareness flagged for quick ecosystem-wide injection
**Decision:** Identified current gap: only Da Vinci's Personal Knowledge gateway injects `{{date}}` into its system prompt; other agents (Jeanne Alter, MERLIN, Midas) lack time/date context. Marked as high-priority, low-effort item for one sitting — inject current date/time into every agent's system prompt using the same pattern, applied ecosystem-wide instead of single-gateway.
**Why:** Time/date awareness is foundational for reasoning about scheduled tasks, recency of information, and temporal context ("what happened yesterday?" vs. "what happened 6 months ago?"). Currently only Da Vinci has it, creating inconsistency where agents operate in different temporal frames. Quick fix with high value.
**Alternatives considered:** Leave as-is (rejected — agents reasoning without temporal context), create a shared `{{date}}` variable in n8n globals (rejected — doesn't solve the per-agent injection problem), add only to Jeanne Alter (rejected — missing the point; do it ecosystem-wide).

### 2026-07-08 - Solomon approved at research stage — overseer/growth agent for Chaldea
**Decision:** New agent concept "Solomon" approved at research/keep-in-view priority (not scheduled for active development yet). Role: ecosystem administrator and growth agent, FGO-lore fit as Chaldea's overseer. Functionality: weekly cron-triggered review of all agents' activity, logs, and decisions; proposes fixes/optimizations for user approval. Explicitly human-in-the-loop — no autonomous self-modification of agents or infrastructure.
**Why:** Current agent ecosystem has no oversight layer to track patterns (e.g., "MERLIN re-discovers the same Cloudflare expiry risk every week instead of remembering it was already reported"). Solomon consolidates meta-analysis across agents without the risk of agents modifying themselves autonomously. Weekly cadence matches natural session frequency.
**Alternatives considered:** Monthly review (rejected — too infrequent for fast-moving issues), autonomous agent self-modification (rejected — too risky, removes user agency), no oversight (rejected — misses systematic improvement opportunities).

### 2026-07-08 - Voice organ (Whisper STT + local TTS) approved at research stage for Jeanne Alter
**Decision:** New organ approved at research/keep-in-view priority: Whisper STT + local TTS for Jeanne Alter. Not prioritized ahead of architecture refactor, but documented as a planned capability. Explicitly deferred vision-model deployment and live browser vision/control due to VRAM contention risk.
**Why:** Voice I/O is lower VRAM footprint than vision models. Current hardware (single RX 6700 XT, 12GB) cannot comfortably run a vision model and the main chat model concurrently — vision is a hard blocker due to GPU memory, not just a priority call. Voice-first investment is safer for current hardware and adds utility for hands-free interaction.
**Alternatives considered:** Vision first (rejected — VRAM risk on single GPU), both in parallel (rejected — same VRAM issue), neither (rejected — both would improve agent ergonomics), cloud-hosted vision (rejected — wants local-first, privacy-first design).
