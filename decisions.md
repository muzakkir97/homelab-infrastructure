### 2026-07-09 - Scathach build requires LangGraph evaluation pre-implementation
**Decision:** Scathach (Lancer-class servant, career growth/research agent, 1st build priority in agent roster as of May 16, 2026) must have LangGraph evaluation completed before implementation scope is defined. Career-research workflows expected to require autonomous multi-step reasoning loops that n8n handles poorly for this use case. LangGraph fit assessment is a pre-requisite for Scathach's build scope definition, not optional pre-work.
**Why:** Current agents (Jeanne Alter, Da Vinci, MERLIN, Midas, EMIYA) fit n8n's strengths (multi-step workflows with external tools, stateful branching). Scathach's intended function (career research with multi-step autonomous reasoning) may require state-graph structure that LangGraph provides better than n8n. Evaluating LangGraph early prevents mid-build discovery that the chosen framework is mismatched to the use case.
**Alternatives considered:** Build Scathach on n8n first, refactor to LangGraph if needed (rejected — wastes time if mismatch is found), skip LangGraph evaluation entirely (rejected — risks building on the wrong framework), use CrewAI instead (rejected — already designated CrewAI for multi-agent discussion, not single-agent reasoning).

### 2026-07-08 - Multi-agent discussion protocol uses CrewAI, not LangGraph
**Decision:** Selected CrewAI as the target framework for future multi-agent discussion capability, superseding an earlier tentative LangGraph approach. Scoped as research priority, deferred from active development. Discussion trigger: complexity-based (quick topics via Telegram, complex/research topics run in background). Disagreement resolution: Jeanne Alter presents both sides; user arbitrates (explicitly not consensus-forced, not auto-decided). Reporting: summary-only by default, full transcript on request.
**Why:** CrewAI's role-based mental model (agents as distinct personas with roles and tools) maps directly onto the existing FGO-persona structure of Chaldea agents, making adoption faster and more coherent than LangGraph's lower-level state-graph approach. CrewAI gets to a working prototype faster. Human-in-the-loop arbitration preserves user agency over agent decisions and avoids the risk of agents forming a consensus that contradicts user intent.
**Alternatives considered:** LangGraph (rejected — more flexible but requires more custom state machine code, steeper learning curve for marginal benefit), multi-turn Jeanne Alter conversation (rejected — single-agent reasoning not suitable for distinct perspectives), force consensus (rejected — removes user arbitration, bad precedent for agent autonomy).

### 2026-07-08 - Nextcloud Deck falls under Da Vinci's exclusive write scope
**Decision:** Nextcloud Deck (Homelab board, ID 4) is added to Da Vinci's documentation write scope, subject to the same "sole writer" discipline as Obsidian. No other agent (including Jeanne Alter, MERLIN, Midas) writes to Deck directly; all board updates flow through Da Vinci's pipeline.
**Why:** Preserves the single-writer-per-storage-system pattern that prevents concurrent write conflicts and maintains clear data provenance. Deck is a kanban view into the same ROADMAP/phase data that Da Vinci already manages, not a separate system. Centralizing writes eliminates merge conflicts and ensures board state stays synchronized with documentation.
**Alternatives considered:** Allow agents to write cards directly (rejected — introduces concurrent-write race conditions, breaks sync), use Deck as read-only reporting surface (rejected — defeats the purpose of a kanban board for work tracking), delegate Deck management to a separate agent (rejected — adds unnecessary coordination overhead).

### 2026-07-08 - Da Vinci documentation pipeline gains 9th step: Nextcloud Deck sync
**Decision:** Da Vinci's update pipeline gains a 9th step that syncs all documentation changes to Nextcloud Deck cards. Trigger: every pipeline run (broadest option). Matching mechanism: hidden `sync-id` tag in card description footer (format: `---\n🔖 sync-id: phase-24.10`). Update behavior: match-and-update existing cards silently for status changes; create new cards only for genuinely new items. Stack routing: status-keyword based (e.g., "Complete" → Done, "In Progress" → In Progress). Scope: Homelab board only (Career/Personal boards excluded for now). Precondition: all ~30 existing Homelab board cards must be manually backfilled with sync-id tags by the user before this automation goes live.
**Why:** Deck sync keeps the kanban board as a real-time view of work state without manual duplication. Broadest-trigger option chosen deliberately despite board-noise risk — better to over-sync than create blind spots where documentation updates silently miss the board. Hidden sync-id tags enable reliable parsing without cluttering human-readable card bodies. Manual backfill chosen over automated "adopt" pass because known board conflicts (duplicate Phase 7E cards, stale completed items still visible, Da Vinci Stage 2 status ambiguity) require human judgment; fuzzy automated matching could create worse conflicts.
**Alternatives considered:** Sync only certain phases (rejected — creates blind spots), always create new cards (rejected — duplicates on every status update), automated backfill (rejected — can't handle known conflicts correctly), store sync-id in card title (rejected — clutters readability).
