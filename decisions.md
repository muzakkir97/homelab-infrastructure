### 2026-07-17 - Pelican web file editor banned for PalWorldSettings.ini; pct exec standardized
**Decision:** Abandoned Pelican's web-based file editor for all future PalWorldSettings.ini edits. Standardized on `pct exec` + `sed`/direct file writes from the Proxmox host instead. Root cause: Pelican's web editor introduces hard line breaks into PalWorldSettings.ini's OptionSettings=(...) block when saving edits (corrupted file observed: 13 lines instead of required 2). Palworld silently ignores the entire OptionSettings block when the block is broken across multiple lines, falling back to defaults for all settings in that block (confirmed: PalCaptureRate change had no in-game effect despite correct file content). Safe editing method: `pct exec <CTID> -- sed -i 's/old/new/g' /path/to/PalWorldSettings.ini` or equivalent direct file writes from Proxmox host. This bypasses Wings and Pelican's file editor entirely. Related: Pelican's PalworldServerConfigParser (run at every container boot) auto-populates PublicIP from local allocation (192.168.30.219) when the egg's "Public IP" variable is empty, causing silent reverts of manual edits; fixed by enabling User Editable + User Viewable permissions on the egg variable and populating it at container creation time.
**Why:** Pelican's web editor is not safe for multi-line ini values. Direct pct exec access to the container filesystem is reliable, preserves file formatting, and bypasses the buggy editor entirely. Prior lesson (avoiding ini edits via Pelican) was now reconfirmed with root-cause detail. This is a documented constraint for all Pelican-managed servers with complex ini structures going forward.
**Alternatives considered:** Delete and regenerate PalWorldSettings.ini from a template (rejected — loses valid edits already made), use Pelican's web editor more carefully (rejected — the tool itself corrupts the format; careful use doesn't prevent the bug), edit via SSH directly (rejected — less reliable than pct exec for container access from Proxmox host).

### 2026-07-17 - Palworld backup method: pct exec + pct pull standardized; panel download bypassed
**Decision:** Established reliable backup method for any Pelican-managed container's live data: `pct exec <CTID> -- tar -czf /tmp/backup.tar.gz -C <path> <folder>` followed by `pct pull <CTID> /tmp/backup.tar.gz <destination>` from the Proxmox host. This bypasses Wings/panel file download entirely and works directly at the container level. Pelican panel's file Download function (Files tab → Archive → Download) has a reproducible "resource not found" 404 error for Palworld; root cause unidentified (suspected Wings node FQDN misconfiguration in signed URL generation), not investigated further due to time-sensitivity of the backup need. Workaround is proven reliable and doesn't require debugging the panel. Palworld SaveGames backup created this session: `/mnt/hdd-backup-1/palworld-backup-20260716/palworld-save-backup.tar.gz` (22M, verified contents).
**Why:** Panel download is broken; direct pct access works every time. For live data that needs immediate backup (player progress at stake), the pct method is faster than debugging signed URLs. This same method applies to any Pelican container on any node, not just Palworld.
**Alternatives considered:** Continue debugging the 404 error (rejected — time-sensitive backup need took priority, and pct method is proven), try different browsers/caching (rejected — issue reproduced even on internal IP, ruling out Cloudflare), request Wings/panel logs for signed URL generation (rejected — pct method works now, can investigate panel issue later if needed).

### 2026-07-17 - Palworld/Terraria restart scheduling to consolidate with nightly vzdump into single maintenance window
**Decision:** Decided to consolidate Palworld and Terraria restart schedules with the nightly vzdump backup job into a single off-peak maintenance block, rather than treating them as separate scheduled events. Current issue: nightly vzdump backup (running ~02:11 AM, sequential across all containers including CT 222 ntfy and CT 223 Langfuse) causes CPU/disk I/O contention with active Palworld gameplay, resulting in observable in-game stutter with 3 concurrent players. Palworld and Terraria are both game servers that need periodic restarts; bundling them with vzdump into a single verified dead-low-usage hour minimizes player impact and clarifies the maintenance window. Exact consolidation window not yet finalized — pending confirmation of a reliably unused hour (current 02:11 AM slot may be acceptable, but requires validation that no players are active at that time and that the consolidated runtime is acceptable). Implementation: reschedule vzdump job via Datacenter → Backup, set up Pelican Schedules for Palworld and Terraria server restarts within the same window.
**Why:** Concentrating all server maintenance into one block (instead of distributed restarts throughout the week) makes it easier for players to understand the maintenance window and plan around it. Keeps stutter-inducing backup I/O isolated from gameplay hours. Current stutter was not a Palworld capacity issue — it was directly caused by concurrent vzdump, so moving vzdump away from gameplay hours solves the actual problem.
**Alternatives considered:** Increase Palworld CPU/memory allocation (rejected — stutter was caused by host-level I/O contention, not Palworld's container resources), keep backup and restart times separate (rejected — misses opportunity to consolidate, leaves players uncertain about when maintenance occurs), move csak backup to a different day (rejected — nightly backup frequency is important for data safety).

### 2026-07-14 - node_exporter /mnt exclusion bug fixed; SATA cable swap deferred
**Decision:** Fixed node_exporter filesystem metric exclusion bug rather than silencing the alert rule. Root cause: Debian's `prometheus-node-exporter` package ships with default `--collector.filesystem.mount-points-exclude` regex containing `mnt`, rendering all `/mnt/*` mountpoints (hdd-backup-1, hdd-backup-2, ssd-storage, pve/kinmoon-smb) invisible to Prometheus since at least 2026-05-16. Alert `MountpointMissing_hddbackup1` was stuck in active state for 8 days (2026-07-06 → 2026-07-14) because the underlying metric was never exposed. Fix: edited `/etc/default/prometheus-node-exporter` to remove `mnt` from the exclude regex (`^/(dev|proc|run|sys|media|var/lib/docker/.+|var/lib/containers/storage/.+)($|/)`), restarted `prometheus-node-exporter.service`, confirmed all three /mnt mountpoints now report metrics and alert auto-resolved. Separately: deferred physical SATA cable/port swap for hdd-backup-1 indefinitely — no budget currently available. User clarified hdd-backup-1 is active primary storage for Nextcloud (not a redundant copy); Kinmoon NAS receives only nightly rsync copy at 03:00, not a live mirror. Prior dmesg SATA link error (2026-07-06) was one-time, self-resolved same day — not an ongoing hardware fault.
**Why:** The alert rule logic was correct; the detection mechanism was broken. Fixing the root cause (the exclusion config) rather than silencing the rule preserves alert legitimacy. Dashboards tied to /mnt mountpoints have been silently empty since May 16 and will now populate correctly. This is a Debian package default that could reappear on future OS upgrades, worth documenting. hdd-backup-1 remains necessary as the live landing zone for Nextcloud and vzdump before any offsite sync; no consolidation onto hdd-backup-2 is needed.
**Alternatives considered:** Silence or delete the alert rule (rejected — rule intent was correct, only the metric exposure was broken), pursue physical SATA cable swap immediately (rejected — no budget, and the one-time error has already self-resolved), move Nextcloud to hdd-backup-2 (rejected — no technical need, hdd-backup-1 is functioning correctly once metrics are visible again), treat the 8-day silence as acceptable (rejected — silent Alertmanager failures mask real infrastructure issues).
